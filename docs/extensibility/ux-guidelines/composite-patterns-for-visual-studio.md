## <a name="data-visualization"></a><a name="BKMK_DataVisualization"></a>데이터 시각화

### <a name="overview"></a>개요
 차트는 의사 결정을 향상시키기 위해 데이터를 집계하고 시각화하는 시각적 방법입니다. 사용자가 많은 데이터에 직면하는 데 도움이 될 수 있지만 주의를 기울여야 할 사항과 조치가 필요한 것을 볼 수 있는 의미는 거의 없습니다.

 다음 조건 중 어느 것이 true인 경우 사용자는 차트의 이점을 누릴 수 있습니다.

- 차트는 사용자가 수행할 수 있는 작업을 식별하는 데 도움이 될 까요?

- 차트를 통해 사용자가 잠재적인 변경의 결과를 예측할 수 있습니까?

- 차트는 사용자가 추세를 발견하고 패턴을 식별하는 데 도움이 될 것인가?

- 차트를 통해 사용자가 더 나은 결정을 내릴 수 있습니까?

- 차트는 사용자가 주어진 맥락에서 가질 수있는 특정 질문에 대답하는 데 도움이 될 것인가?

#### <a name="general-rules-for-charts"></a>차트에 대한 일반 규칙

- 데이터에 레이블을 명확하게 지정합니다. 설명이없는 일러스트는 예쁜 사진일 뿐입니다.

- 비율이 왜곡되지 않도록 하려면 축을 0으로 시작합니다. 줄 길이와 막대 크기는 데이터 점 간의 관계를 이해하는 데 중요한 시각적 단서입니다.

- 인포그래픽이 아닌 차트를 만듭니다. 인포그래픽은 데이터의 예술적 표현이며, 주요 목표는 시각적 스토리텔링입니다. 차트는 시각적으로 매력적일 수 있지만 데이터 자체를 말하게 할 수 있습니다.

- skeumorphism, 그림 막대 그래프, 대비 해시 마크 및 기타 인포 그래픽 터치를 피하십시오.

- 3D 효과를 장식 요소로 사용하지 마십시오. 사용자가 정보를 이해할 수 있는 능력에 진정으로 필수적인 경우에만 사용하십시오.

- 두 개 이상의 색상이 이러한 유형의 차트를 올바르게 읽고 해석하기 어렵게 만들 수 있기 때문에 여러 줄과 채우기를 사용하지 마십시오.

- 차트(또는 그림)를 개념을 이해하거나 데이터와 상호 작용하는 유일한 수단으로 사용하지 마십시오. 이것은 시각 장애가있는 사용자에게 어려움을 안겨주게합니다.

- 페이지에 무상 또는 장식 요소로 차트를 사용하지 마십시오. 즉, 차트에 값을 추가하지 않거나 사용자가 문제를 해결하는 데 도움이 되지 않는 경우 사용하지 마세요.

### <a name="chart-types"></a>차트 종류
 Visual Studio에서 사용되는 차트 유형에는 막대 차트, 꺾은줄 차트, 링 차트 또는 "도넛 형 차트"로 알려진 수정된 원형 차트, 타임라인, 분산형 차트("클러스터 차트"라고도 함) 및 Gantt 차트가 포함됩니다. 각 차트 유형은 서로 다른 유형의 정보를 전달하는 데 유용합니다.

## <a name="on-object-ui-and-peeking"></a><a name="BKMK_OnObjectUI"></a>개체 내 UI 및 엿보기
 이 섹션에서는 Visual Studio에 고유한 개체 UI 유형인 코드 엿보기보기라고도 하는 엿보기에 대한 컨텍스트를 제공합니다.

### <a name="overview"></a>개요

- 개체 내 UI는 사용자에게 기본 작업을 손상시키지 않으면서 더 많은 정보 나 상호 작용을 제공해야합니다.

- Visual Studio의 개체 UI에 대한 기본 패턴을 "관심 지점의 정보"라고 합니다.

- Visual Studio의 개체 UI는 인라인 또는 부동 이며 내구성 또는 일시적입니다.

  - Visual Studio의 개체 에 UI 유형인 코드 엿보기는 인라인이며 내구성이 뛰어납니다.

  - 비주얼 스튜디오에서 개체 UI의 유형인 CodeLens가 부동 및 일시적입니다.

  코드 의 작동 방식을 이해하거나 해당 코드에 대한 세부 정보를 찾는 경우 개발자가 컨텍스트를 전환하고 다른 콘텐츠 또는 다른 창으로 이동해야 하는 경우가 많습니다. 사용자가 기본 창을 벗어나면 원래 작업에 집중할 수 있기 때문에 이러한 컨텍스트 이동은 중단될 수 있습니다. 또한 창을 전환하면 다른 UI에 의해 원래 코드가 가려지는 경우 원래 컨텍스트를 되돌리기가 어려울 수 있습니다.

  개체 UI는 "관심 지점의 정보"라는 패턴을 따릅니다. 이러한 메시지, 팝업 창 및 대화 상자는 사용자에게 주요 작업에 집중하지 않고 설명또는 상호 작용을 추가하는 추가 관련 정보를 제공합니다. 개체 UI의 예로는 사용자가 알림 영역의 아이콘 위로 포인터를 마우스로 가리키면 나타나는 팝업 창, 맞춤법이 틀린 단어 아래의 빨간색 물결선 및 Visual Studio 2013에 도입된 엿보기가 있습니다.

### <a name="decision-points"></a>결정 사항
 Visual Studio에는 주의 지점에서 이 정보 패턴을 사용하는 여러 가지 방법이 있습니다. 올바른 메커니즘을 선택하고 일관되고 예측 가능한 방식으로 구현하는 것은 전반적인 경험에 필수적입니다. 그렇지 않으면 사용자에게 콘텐츠 자체의 초점을 방해하는 혼란스럽거나 일관성 없는 환경이 제공될 수 있습니다.

#### <a name="relationships-between-master-and-detail-content"></a>마스터 콘텐츠와 세부 콘텐츠 간의 관계
 주의 시점의 정보는 사용자가 초점을 맞춘 콘텐츠("마스터" 콘텐츠)와 추가 관련 콘텐츠("세부 정보" 콘텐츠) 간의 관계를 표시하는 데 사용됩니다. 이 패턴에서, 상세 콘텐츠는 사용자가 작업하는 콘텐츠와 명확하게 관련이 있으며 마스터 콘텐츠에 가깝게 표시될 수 있다. 마스터 콘텐츠를 압도하지 않고 요약할 수 없는 추가 정보 나 정보는 도구 창과 같은 다른 패턴을 따라야 합니다.

- **항상** 마스터 콘텐츠에 가까운 거리에서 상세 내용을 표시합니다.

- **항상** 세부 정보 콘텐츠를 통해 사용자가 마스터 콘텐츠에 집중할 수 있도록 해야 합니다. 이를 달성하는 가장 좋은 방법은 세부 콘텐츠를 마스터 콘텐츠에 최대한 가깝게 렌더링하는 것입니다. 이 작업은 마스터 콘텐츠 옆의 팝업 창에서 상세 콘텐츠를 렌더링하거나 마스터 콘텐츠 아래에 세부 콘텐츠 인라인을 렌더링하여 수행할 수 있습니다.

- 사용자가 마스터 콘텐츠에서 멀어지는 주의 지점에서 정보를 사용하지 **마십시오.** 사용자가 세부 정보 콘텐츠를 별도로 확인해야 하는 경우 사용자가 이 작업을 수행할 수 있는 명시적 작업을 노출합니다.

#### <a name="design-details"></a>디자인 세부 정보
 개체 UI가 올바른 선택이라고 판단되면 다음 네 가지 주요 디자인 고려 사항이 있습니다.

1. **지속성:** 콘텐츠가 내구성이 뛰어나거나 일시적일 것으로 예상되나요?
   사용자가 참조하거나 상호 작용할 수 있도록 정보를 계속 표시하시겠습니까? 또는 사용자가 정보를 빠르게 한 눈에 본 다음 주요 작업을 계속할 까요?

2. **콘텐츠 유형:** 콘텐츠가 정보, 실행 가능 또는 탐색이 가능합니까?
   사용자가 마스터 콘텐츠에 대한 추가 세부 정보가 필요합니까? 사용자가 마스터 콘텐츠에 영향을 주는 작업을 완료해야 합니까? 또는 사용자가 다른 리소스로 이동해야 합니까?

3. **표시기 유형:** 주변 표시등이 의미가 있습니까?
   정보를 유용한 방법으로 요약하고 마스터 콘텐츠를 압도하지 않고 표시 할 수 있습니까?

4. **제스처:** UI를 호출하고 해제하는 데 어떤 제스처가 사용됩니까?
   사용자가 세부 정보 콘텐츠를 가져오고 어떻게 보내나요? 일시적 상태와 지속 상태를 전환하는 고정과 같은 제스처를 추가하는 데 가치가 있습니까?

   이러한 네 가지 결정 지점각각은 개체 UI의 주요 구성 요소에 영향을 미칩니다.

### <a name="on-object-ui-components"></a>개체 내 UI 구성 요소

    - 실행 가능: 마스터 콘텐츠를 변경하는 명령

    - 탐색: 사용자를 다른 창또는 응용 프로그램(예: MSDN)으로 연결하는 링크

3. 제스처

    - 호출

    - 해임

    - 핀 고정

    - 기타 상호 작용

4. 지속성 및 커밋 모델

    - 임시

    - 지속성

    - 자동

    - 요청 시

5. 주변 표시기(선택 사항)

    - 물결선 밑줄

    - 스마트 태그 아이콘

    - 기타 주변 표시기

#### <a name="container-content-presenter-type"></a>컨테이너(콘텐츠 발표자) 유형
 관심 의 시점에서 콘텐츠를 제시하는 데 사용할 수있는 두 가지 주요 옵션이 있습니다 :

1. **인라인:** Visual Studio 2013 코드 편집기에서 소개된 엿보기와 같은 인라인 발표자는 기존 콘텐츠를 이동하여 새 콘텐츠를 위한 공간을 만듭니다.

    - 사용자가 현재 존재하는 콘텐츠를 참조하거나 상호 작용하는 데 상당한 시간을 할애할 것으로 예상되는 경우 인라인 발표자를 **선호합니다.**

    - 사용자가 제시한 정보를 한눈에 보고 싶을 것으로 예상되는 경우 인라인 발표자를 **피한** 다음 중단을 최소화하면서 주요 작업을 계속하십시오.

2. **부동:** 부동 발표자는 선택한 콘텐츠에 최대한 가깝게 배치되지만 기존 콘텐츠의 레이아웃은 변경하지 않습니다. 선택된 심볼에 가장 가까운 공백 위에 부동 콘텐츠 패널을 표시하는 등 다양한 전략을 사용할 수 있습니다.

    - 사용자가 현재 제시한 정보를 한눈에 보고 싶을 것으로 예상되는 경우 부동 발표자를 **선호한** 다음 중단을 최소화하면서 주요 작업을 계속하십시오.

    - 사용자가 현재 존재하는 콘텐츠를 참조하거나 상호 작용하는 데 상당한 시간을 할애할 것으로 예상되는 경우 부동 발표자를 **사용하지 마십시오.**

#### <a name="content-type"></a>내용 유형
 개체 내 UI 컨테이너 내에 표시할 수 있는 세 가지 주요 콘텐츠 유형이 있습니다. 이러한 유형의 정보의 조합을 표시할 수 있습니다. 세 가지 유형은 다음과 같습니다.

1. **정보:** 대부분의 개체 UI 컨테이너는 일종의 정보 콘텐츠를 표시합니다. 콘텐츠는 환경의 현재 상태에 대한 정보를 나타낼 수 있거나 환경의 잠재적미래 상태에 대한 정보를 나타낼 수 있습니다. 예를 들어 리팩터링과 같은 특정 명령이 기존 코드에 미치는 영향을 표시하는 데 사용할 수 있습니다.

    - **항상** 표시되는 정보의 표준 표현을 사용합니다. 예를 들어 코드는 구문 강조 표시가 완료된 코드처럼 표시되어야 하며 사용자가 설정한 글꼴 및 기타 환경 설정을 준수해야 합니다.

    - **동일한** 정보가 마스터 콘텐츠로 표시되는 경우 가능한 정보 콘텐츠에 대한 작업을 항상 지원하는 것이 좋습니다. 예를 들어 개체 내 UI 컨테이너 내에 기존 코드를 표시하는 경우 해당 코드를 찾아보고 수정하는 기능을 지원하는 것이 좋습니다.

    - 잠재적인 미래 상태를 나타내는 정보 콘텐츠를 표시하는 경우 **항상** 다른 배경색을 사용하는 것이 좋습니다.

2. 실행 가능: 일부 개체 내 UI 컨테이너는 리팩터링 작업을 수행하는 등 마스터 콘텐츠에 대한 일부 작업을 수행할 수 있는 기능을 제공합니다.

    - **항상** 정보 콘텐츠와 별도로 실행 가능한 명령을 배치합니다.

    - 적절한 경우 **항상** 작업을 활성화하고 사용하지 않도록 설정합니다.

    - **항상** 대화 상자 내부의 명령을 나타내는 표준 지침을 참조하십시오.

    - **항상** 개체 내 UI 컨테이너에 노출되는 작업 수를 최소한으로 유지합니다. 개체 내 UI와 상호 작용하는 것은 가볍고 빠른 경험이어야 합니다. 사용자는 개체 내 UI 컨테이너 자체를 관리하는 데 에너지를 소비할 필요가 없습니다.

    - **개체** 내 UI 컨테이너를 닫거나 해제하는 방법과 시기를 항상 고려하십시오. 가장 좋은 방법은 마스터와 세부 정보 콘텐츠 간의 대화 상자를 마무리하는 모든 작업은 해당 작업이 호출될 때 개체 내 UI 컨테이너도 닫아야 합니다.

3. **탐색:** 일부 개체 UI 컨테이너에는 사용자의 웹 브라우저에서 MSDN 아티클을 여는 것과 같이 사용자를 다른 창이나 응용 프로그램으로 연결하는 링크가 포함됩니다.

    - **사용자가** 다른 콘텐츠로 이동하여 놀라지 않도록 항상 "열기"로 탐색 링크를 준비합니다.

    - **항상** 탐색 링크를 실행 가능한 링크와 분리합니다.

#### <a name="ambient-indicators-optional"></a>주변 표시기(선택 사항)
 주변 표시기는 코드의 나머지 부분에서 대조적인 색상으로 표시되는 텍스트를 포함하여 미묘하거나 물결선 밑줄 및 스마트 태그 아이콘과 같은 간지럼 기호를 포함하여 분명할 수 있습니다. 주변 표시기는 추가 관련 정보의 가용성을 전달합니다. 이상적으로는 사용자가 상호 작용할 필요 없이 유용한 정보를 제공하는 것이 좋습니다.

- **항상** 사용자를 방해하거나 압도하지 않도록 주변 표시기를 배치하십시오. 이러한 방식으로 주변 표시기를 배치하는 것이 불가능한 경우 다른 솔루션을 고려하십시오.

- **항상** 주변 표시기를 관련된 콘텐츠에 가능한 한 가깝게 배치합니다.

- **항상** 사용할 수 있는 정보를 요약하는 표시기를 만들어 보십시오. 사용 가능한 데이터 항목 수(예: 단순히 "참조"가 아닌 "참조 3개")를 제공하거나 데이터를 요약하는 다른 방법을 생각해 보십시오.

  - 표시기의 데이터를 항상 계산하고 표시할 수 없는 경우 값이 계산될 때 즉시 점진적 피드백을 제공하는 것이 좋습니다. 예를 들어 Windows Phone의 전자 메일 라이브 타일이 읽지 않은 전자 메일 수가 증가함에 따라 새로 고치는 방식과 유사하게 사용 가능한 데이터에 대한 업데이트를 반영하는 변경 내용을 애니메이션하는 것이 좋습니다.

- 사용자가 특정 콘텐츠에 대해 합리적으로 받아들일 수 있는 것보다 더 많은 지표를 **추가하지 마십시오.** 주변 표시기는 사용자의 상호 작용없이 유용해야합니다. 지표는 오버플로 및 기타 관리 제어가 필요한 경우 분위기를 잃게 됩니다.

#### <a name="gestures"></a>제스처
 사용자가 마스터 콘텐츠에 집중할 수 있도록 하는 주요 측면은 추가 세부 콘텐츠의 열기 및 해제에 올바른 제스처를 지원하여 사용하는 것입니다.

- **항상** 추가 콘텐츠를 열려면 사용자가 일부 명시적 제스처를 수행하도록 요구합니다. 일반적인 개방형 제스처는 다음과 같습니다.

  - **가리키기:** 도구 설명 또는 비대화형 정보 콘텐츠

  - **명시적 명령:** 인라인 발표자

  - **주변 표시기를 두 번 클릭합니다.** 코드 렌즈 팝업 창

- 사용자가 Esc 키를 누를 때마다 **항상** 세부 정보 콘텐츠를 해제합니다.

- **항상** 개체 에 UI의 컨텍스트를 고려 합니다. 컨테이너 내에서 상호 작용을 허용하는 콘텐츠 발표자의 경우 사용자의 워크플로에 방해가 될 수 있는 호버에 추가 정보를 표시할지 신중하게 고려합니다.

- 편집 가능한 것으로 보이거나 사용자 상호 작용을 초대하는 가리키기 위에 콘텐츠를 **표시하지 마십시오.** 이 동작은 도구 설명의 표준 동작은 커서가 더 이상 커서를 생성한 마스터 콘텐츠 위에 없을 때 즉시 해제하기 때문에 세부 정보 콘텐츠 위로 커서를 이동하려고 하면 사용자를 좌절시킬 수 있습니다.

## <a name="selection-models"></a><a name="BKMK_SelectionModels"></a>선택 모델

### <a name="overview"></a>개요
 선택 모델은 사용자 인터페이스 내에서 하나 이상의 관심 있는 개체에 대한 작업을 나타내고 확인하는 데 사용되는 메커니즘입니다. 이 항목에서는 Visual Studio 문서 편집기(텍스트 편집기, 디자인 표면 및 모델링 표면)에서 선택 상호 작용 패턴에 대해 설명합니다.

 사용자는 작업 중인 작업을 Visual Studio에 표시하는 방법이 있어야 하며 Visual Studio는 작업 중인 작업에 대한 피드백을 사용자에게 예측 가능하게 응답해야 합니다. 사용자와 사용자 인터페이스 간의 차이 또는 잘못된 통신으로 인해 사용자가 작업을 알아차리지 못하여 의도하지 않은 결과가 발생할 수 있습니다. 사용자가 누락되었거나 변경된 것을 볼 때까지 오류가 발견되지 않는 경우가 많습니다. 따라서 선택 모델은 사용자 인터페이스 디자인에서 가장 중요한 부분 중 하나입니다. Visual Studio의 선택 모델은 Windows와 일치하지만 약간의 차이가 있습니다.

 Visual Studio에서 Windows와 마찬가지로 선택 모델은 상호 작용이 발생하는 컨텍스트에 따라 다릅니다. 선택은 다음 네 가지 유형의 개체에서 발생할 수 있습니다.

- 텍스트

- 그래픽 개체

- 목록 및 트리

- 표

  이러한 개체 에는 세 가지 유형의 선택 항목이 있습니다.

- 연속

- 연결이 끊긴

- 지역

#### <a name="scope"></a>범위
 선택의 가장 중요한 구성 요소는 사용자가 작업 중인 창(활성화)과 포커스가 있는 위치(선택)를 알 수 있도록 하는 것입니다. Visual Studio는 Windows에서 창 관리 기능을 확장하지만 정품 인증 체계는 동일합니다. Visual Studio에는 활성화를 위한 두 가지 지표가 있습니다.

 문서 창의 경우 활성 창은 앞면에 오고 배경색을 변경하는 문서 창 탭으로 표시됩니다.

 ![Visual Studio에서 활성 탭 선택](../../extensibility/ux-guidelines/media/0713-01_activetab.png "01_ActiveTab")

 **활성 탭 선택**

 도구 창의 경우 활성 창은 도구 창의 제목 표시줄 영역의 색상 변경으로 표시됩니다.

 ![Visual Studio에서 활성 도구 창 선택](../../extensibility/ux-guidelines/media/0713-02_activetoolwindow.png "02_ActiveToolWindow")

 **노드의 기본 선택을 보여주는 활성 도구 창**

 ![Visual Studio에서 비활성 도구 창 선택](../../extensibility/ux-guidelines/media/0713-03_inactivetoolwindow.png "03_InactiveToolWindow")

 **비활성 공구 창, 노드의 잠재 선택 표시**

 창이 활성화되면 지침의 이 섹션에 설명된 선택 모델에 따라 해당 포커스가 표시됩니다.

#### <a name="context"></a>Context
 Visual Studio는 사용자가 작업하는 위치를 추적하면서 강력한 컨텍스트 개념을 유지하도록 설계되었습니다. 도구 창이든 문서 창이든 하나의 창만 활성화되어 있습니다. 그러나 맨 위 문서 창은 항상 잠재 선택을 유지합니다. 포커스가 도구 창에 있을 수 있지만 마지막으로 활성 상태였던 문서 창에는 비활성 상태에서도 선택 항목이 표시됩니다. 이는 편집한 문서에서 사용자의 컨텍스트를 유지하여 Visual Studio가 도구 창과 문서 창 간에 원활하게 반환하고 이동할 수 있도록 상태를 유지되었음을 보여 주는 것입니다.

### <a name="text-selection"></a>텍스트 선택
 기본 제공 텍스트 편집기와 같이 엄격하게 텍스트인 Visual Studio 편집기는 MSDN의 Windows 사용자 환경 상호 작용 지침의 [마우스 및 포인터](/windows/desktop/uxguide/inter-mouse) 페이지에 설명된 것과 동일한 텍스트 선택 모델 및 모양을 사용합니다. 텍스트 편집기의 입력 포커스는 삽입 점이라는 세로 막대로 표시됩니다. 삽입 점은 단일 픽셀 두께이며 뒤에 나타나는 모든 것의 역으로 색이 지정됩니다. 제어판에서 **키보드** 애플릿의 **속도** 탭에서 **커서 깜박임 속도** 설정에 의해 설정된 속도에 따라 깜박입니다.

#### <a name="contiguous-and-disjoint-selection"></a>연속 및 분리 선택
 텍스트 편집기 내의 선택은 연속된 경우에만 선택됩니다. 분리 텍스트 선택은 허용되지 않지만 그래픽 개체 편집기에서 해결해야 합니다. 사용자의 마우스 포인터가 텍스트 영역 위에 있으면 커서가 I-빔으로 변경됩니다. 한 번의 클릭으로 텍스트 편집기의 삽입 점을 클릭 위치에 배치합니다. 마우스 버튼을 길게 누르면 선택 강조 표시가 시작되고 마우스 버튼을 놓면 선택 강조 표시가 끝납니다.

#### <a name="region-selection-box-selection"></a>영역 선택(상자 선택)
 Visual Studio는 텍스트 편집기에서 영역 선택을 지원하며 이를 상자 선택이라고 합니다. 상자를 선택하면 일반 텍스트 스트림을 따르지 않는 텍스트 영역을 선택할 수 있습니다. 표준 텍스트 선택과 마찬가지로 선택 영역은 연속적이어야 합니다. 상자 선택은 마우스로 드래그하는 동안 Alt 키를 누르고 대기하여 시작됩니다. 상자 선택은 화살표 키를 사용하여 선택 영역을 나타내는 동안 Alt 및 Shift 키를 누루면 시작할 수도 있습니다. 상자 선택은 법선 선택 강조 표시를 사용하고 선택 영역의 끝에서 삽입 점 커서가 깜박이는 것을 표시합니다.

 ![비주얼 스튜디오에서 지역 &#40;박스&#41; 선택](../../extensibility/ux-guidelines/media/0713-04_boxselection.png "0713-04_BoxSelection")

 **비주얼 스튜디오에서 영역(상자) 선택**

#### <a name="text-selection-appearance"></a>텍스트 선택 모양
 편집기에서 활성 및 비활성 선택에 사용되는 색상을 사용자 지정할 수 있습니다. 편집기의 시각적 모양을 사용자 지정하려면 사용자는 **도구 > 옵션으로**이동한 다음 **환경 > 글꼴 및 색상 > 텍스트 편집기**에서 볼 수 있습니다.

### <a name="graphical-selection"></a>그래픽 선택

#### <a name="interaction"></a>상호 작용
 그래픽 오브젝트 선택은 복잡할 수 있으며 다음과 같은 여러 요인에 따라 달라집니다.

- **편집자의 기본 선택 모델입니다.** 그래픽 개체가 포함된 편집기를 사용하여 텍스트 또는 그리드를 편집할 수도 있습니다. 예를 들어 편집기는 Visual Studio XAML 디자이너와 같은 그래픽 개체의 배치를 지원하는 텍스트 기반 편집기일 수 있습니다. 여러 개체 유형을 지원하면 사용자가 다양한 유형의 개체로 구성된 그룹을 선택하는 방법에 영향을 줄 수 있습니다.

- **기본 및 보조 선택 상태에 대한 지원입니다.** 편집기는 객체를 한식으로 편집하고, 서로 정렬하고, 함께 크기를 조정하는 등의 방식으로 개체를 편집할 수 있도록 기본 및 보조 선택 상태를 제공할 수 있습니다.

- **현재 내 편집 지원.** 편집자는 그래픽 개체의 내용을 편집할 수도 있습니다. 예를 들어 사각형 셰이프에는 사용자가 변경할 수 있는 내부 텍스트도 포함될 수 있습니다. 또한 해당 텍스트의 가운데를 배치하거나 양쪽 맞춤이 될 수 있습니다. 내부 편집에는 보다 자세한 수준의 사용자 상호 작용이 포함되므로 사용자에게 상태 정보를 표시하기 위한 적절한 시각적 단서 집합이 필요합니다.

|선택한 객체를 클릭합니다.|객체가 지원하는 경우 현재 편집을 활성화합니다. 오브젝트 외부를 클릭하면 내재 편집 모드가 비활성화됩니다.|

|객체 선택|개체의 경계를 강조 표시하기 위해 테두리, 점선 또는 기타 시각적으로 고유한 처리로 표시됩니다.|

|입력|결과|

|탭|편집기에서 개체의 논리적 순서 간에 포커스 표시기를 이동합니다. 이는 **TabIndex(또는** 이에 상응하는) 속성 값, 개체 만들기 순서 및 편집기의 전반적인 목적에 따라 왼쪽에서 오른쪽 또는 위에서 아래로 할 수 있습니다. Shift+Tab은 초점 표시기의 방향을 반대로 합니다.|

|Ctrl+Alt+마이너스 기호|확대/축소 계수를 한 단계 줄입니다.|
|Ctrl+Alt+플러스 기호|확대/축소 계수를 한 단계 늘립니다.|
|시프트 또는 Ctrl|선택 그룹에 객체를 추가합니다. 또한 Ctrl을 사용하면 선택 그룹에서 개별적으로 개체를 제거할 수 있습니다.|
|Enter|개체에 대한 기본 명령을 수행합니다(일반적으로 열기 또는 편집).|
|F2|객체에 대한 현재 편집을 활성화합니다.|
|화살표 키|선택한 오브젝트를 작은 단위로 누른 화살표 키 방향으로 이동합니다(예: 한 번에 1픽셀).|

|시프트+화살표 키|선택한 개체의 크기를 각 방향으로 작은 증분(예: 한 번에 1픽셀) 크기 조정|
|Ctrl+시프트+화살표 키|선택한 개체의 크기를 각 방향으로 더 큰 증분(예: 한 번에 10픽셀) 크기 조정|

 사용자가 컨트롤을 제자리에서 편집하면 개체가 사용자 입력을 사용하여 자동으로 크기를 조정하는 것이 적합할 수 있습니다. 예를 들어 사용자가 레이블 컨트롤을 편집하는 경우 사용자가 방금 입력한 텍스트를 표시하도록 레이블이 커져야 합니다. 이렇게 하지 않으면 사용자는 텍스트를 편집한 후 컨트롤의 크기를 수동으로 조정해야 합니다. 사용자가 컨트롤을 많이 하는 경우 이 작업은 부패하고 비생산적인 작업이 됩니다.

#### <a name="graphical-containers"></a>그래픽 컨테이너
 경우에 따라 그래픽 편집기는 Windows 양식 패널 컨트롤 또는 HTML 디자이너의 그리드 레이아웃 컨트롤과 같은 다른 그래픽 개체에 대한 컨테이너를 제공합니다. 편집기에서 다른 그래픽 개체에 대한 컨테이너를 제공하는 경우 다음 선택 모델을 컨테이너에만 사용해야 합니다(컨테이너 내의 개체는 위에서 설명한 표준 모델을 따릅니다).

|입력|결과|
|-----------|------------|
|컨테이너를 한 번 클릭|포함된 개체를 직접 선택하지 않고 컨테이너 개체를 선택합니다. 컨테이너는 표준 마우스 및 키보드 입력(위에서 설명한 대로)으로 이동 및/또는 크기를 조정할 수 있습니다. 포함된 객체는 컨테이너를 기준으로 이동되지만 포함된 객체는 직접 선택되지 않는 한 크기가 조정되지 않습니다.|
|컨테이너의 경계 영역 위로 마우스를 가져가기|마우스를 이동 커서로 변환하여 컨테이너를 이동할 수 있음을 나타냅니다.|
|컨테이너의 경계 영역 끌기|마우스를 이동 커서로 변경하고 컨테이너(및 포함된 개체 내)를 이동합니다. 한 번의 클릭으로 먼저 선택하지 않고는 컨테이너를 이동할 수 없습니다.|
|컨테이너 내의 개체를 한 번 클릭|컨테이너를 선택 해제하고(선택한 경우) 클릭한 개체만 선택합니다.|
|시프트 +를 클릭 OR Ctrl +포함 된 개체 및 / 또는 컨테이너를 클릭|클릭한 객체를 기존 선택 또는 선택 그룹에 추가합니다. 클릭한 객체가 이미 선택 그룹의 구성원인 경우 선택 그룹에서 제거됩니다.|

 포함된 객체는 이전 단면에 설명된 기본 선택 모델을 준수해야 합니다. Windows Forms 디자이너의 유용성 테스트에서 사용자는 포함 된 개체에 대한 원활한 액세스를 중간 단계(포함 개체에 의해 부과됨)하지 않고 원활하게 액세스할 수 있습니다.

### <a name="view-selection-models"></a>선택 모델 보기

#### <a name="tree-view"></a>트리 보기
 트리 뷰의 선택 영역은 간단한 강조 표시로 표시됩니다. 사용자가 노드 이름 또는 노드 아이콘을 클릭하면 노드가 선택됩니다. 노드 왼쪽의 삼각형 글리프는 트리 컨트롤을 확장하거나 축소하지만 한 가지 예외를 제외하고 는 선택 영역이 해당 노드의 자식에 있을 때 부모 노드를 축소하면 선택 영역이 부모로 이동합니다.

 ![Visual Studio에서의 일반적인 트리 뷰](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713-25_TreeView")

 **Visual Studio에서의 일반적인 트리 뷰**

 트리 뷰는 트리의 여러 수준에서도 연속 및 분리 선택을 지원할 수 있습니다. 보이는 트리 노드에서 연속 또는 분리된 여러 선택을 수행해야 합니다. 노드가 축소되면 분리 선택이 손실되고 축소된 노드가 선택 영역을 가져옵니다. 이러한 방식으로 사용자는 작업의 영향을 받는 노드를 볼 수 있습니다. 노드가 축소되면 영향을 받을 수 있는 노드가 명확하지 않게 됩니다.

 부모 노드를 선택하면 작업이 부모 노드와 모든 자식에 적용되는 것이 타임이 있을 수 있지만 작업이 부모에 적용되어야 합니다. 이 경우 작업 중에 사용자에게 "모든 자식에 적용" 옵션을 명시적으로 지정하기 위해 확인란 또는 확인 대화 상자와 같은 추가 UI를 제공합니다.

##### <a name="renaming"></a>이름 바꾸기
 트리의 노드가 이름 바꾸기를 지원하는 경우 이름을 변경해야 합니다. 내부 작업은 Visual Studio의 모든 트리 컨트롤에서 표준이어야 합니다. 사용자 입력을 수락할 준비가 된 노드의 전체 이름을 포함하는 텍스트 선택과 함께 즉시 장소 편집 모드를 활성화하는 이름 바꾸기 명령을 제공합니다. 노드가 파일을 나타내는 경우 파일 이름에는 확장명이 포함되어야 합니다. 선택 강조 표시영역에는 확장이 아닌 파일 이름의 본문만 포함되어야 합니다.

|입력|결과|
|-----------|------------|
|Enter 키|이름 바꾸기 작업을 커밋합니다.|
|Esc 키|이름 바꾸기 작업 취소|
|내면 편집 영역 외부 클릭|이름 바꾸기 작업을 커밋합니다.|
|실행 취소|이름 바꾸기 작업을 취소하기 쉬운 취소 제공|

#### <a name="selection-within-lists-and-grid-controls"></a>목록 및 그리드 컨트롤 내에서 선택
 목록 선택의 주요 개념은 행 기반이며, 이는 선택 영역이 선택될 때 전체 행이 단위로 선택된다는 것을 의미합니다. 반대로 그리드는 행의 다른 측면에 영향을 주지 않고 특정 셀을 선택할 수 있습니다. 그리드에는 상위 행과 상호 작용하여 계층의 전체 분기를 선택하고 선택을 취소할 수 있는 중첩행(예: TreeGrid)의 계층 구조도 포함될 수 있습니다. 목록의 선택은 전체 데이터 행에 간단한 강조 표시 색상으로 표시됩니다. 포커스는 현재 편집 가능한 행 또는 셀 주위의 단일 픽셀 점선 테두리로 표시됩니다(모든 셀이 읽기 전용인 경우 행).

> [!NOTE]
> **포커스와** **선택은** 서로 다른 개념입니다. *포커스는* 다른 개체를 명시적으로 지시하지 않는 입력을 수신하도록 타겟팅되는 UI 요소를 나타내며, *선택은* 후속 작업이 발생할 수 있는 개체 집합에 개체가 포함된 상태를 나타냅니다.

 목록에서 선택한 항목은 연속, 분리 또는 영역일 수 있습니다. 여러 선택이 허용되는 경우 연속 및 분리 선택이 항상 지원되어야 하지만 영역(상자) 선택 항목에 대한 지원은 선택 사항입니다. 영역 선택은 목록 본체의 공백을 드래그하여 시작됩니다.

| Object | 선택 |
|--------|------------|
| 목록 | 연속 |
| 목록 | 연결이 끊긴 |
| 목록 | 지역 |

 목록에서 한 번 클릭하면 클릭이 발생한 행이 선택됩니다. 사용자가 현재 편집을 지원하는 목록 셀을 클릭하면 셀도 즉시 활성화되어 즉시 즉시 작동됩니다. 그렇지 않으면 전체 행이 즉시 선택되고 강조 표시가 표시됩니다.

 목록 본문에서 드래그하는 것은 다음 세 가지 중 하나를 수행합니다.

- 목록이 이를 지원하고 마우스 다운이 공백에 있는 경우 영역 선택을 시작합니다.

- 목록 셀 또는 행이 끌기 소스가 되는 것을 지원하는 경우 끌기/놓기 작업이 시작됩니다.

- 현재 행을 선택합니다.

##### <a name="in-place-editing"></a>현재 장소 편집
 현재 편집이 허용되면 간단한 편집 제어와 속성 선택기의 두 가지 기본 모델이 있습니다. 간단한 편집 컨트롤을 사용하면 콘텐츠가 강조 표시되고 현재 사용 가능한 편집이 활성화되는 즉시 사용자 입력이 가능합니다. 속성 선택기를 구현하는 경우 속성 선택기를 호출하는 단추는 현재 선택 영역이 활성화되면 표시되고 현재 선택 영역은 강조 표시되지 않습니다. 선택기 버튼은 셀에서 오른쪽 맞춤이어야합니다. 현재 작업 중 편집 예제는 Visual Studio의 **속성 창** 및 **작업 목록을** 참조하세요.

##### <a name="keyboard-support"></a>키보드 지원
 목록 및 그리드에서 선택하기 위한 키보드 지원은 표준 Windows 규칙을 따릅니다.

- 화살표 키는 포커스가 이동될 때 각 행/셀을 선택하여 목록을 탐색합니다.

- Shift + 화살표는 화살표 키 방향으로 연속 선택을 수행합니다.

- Ctrl + 화살표 다음에 스페이스바가 선택 영역에서 목록 항목을 추가하고 제거하는 사이에 전환하여 분리 선택을 만듭니다.

- 중첩계층이 포함된 그리드의 경우 오른쪽 화살표 키는 부모 행을 확장하고 왼쪽 화살표 키는 하나를 축소합니다.

- 탭 키는 셀을 편집할 수 있는 경우 현재 행의 셀 간에 포커스를 이동합니다.

- Enter 키는 목록의 항목에 대한 기본 명령을 수행합니다(종종 **열기).**

- F2 키는 현재 선택한 셀에 대한 현재 편집을 활성화합니다.

## <a name="persistence-and-saving-settings"></a><a name="BKMK_PersistenceAndSavingSettings"></a>지속성 및 저장 설정

### <a name="overview"></a>개요
 Visual Studio의 각 소프트웨어 구성 요소는 일반적으로 자체 상태 및 지속성을 담당하지만 Visual Studio는 창 크기 및 위치와 같은 경우에 따라 설정을 자동으로 저장합니다. 아래 표는 자동으로 저장된 설정과 명시적 사용자 또는 프로그래밍된 작업을 수행해야 하는 설정의 조합입니다.

|Object|저장해야 할 것|저장 시기|저장 위치|

- 사용자가 대화 상자 또는 창 내에서 위치를 변경하는 경우

- 사용자가 포커스를 다른 창으로 전송하는 경우

- 사용자가 설계에서 디버그 모드로 전환하는 경우

- 사용자가 자신의 계정을 로그오프할 때

- 컴퓨터가 최대 절전 모드로 전환되거나 종료될 때

- 컴퓨터/하드 드라이브를 다시 포업하고 다시 설정하려고 할 때

### <a name="window-configurations"></a>창 구성
 창 구성은 개발 환경의 기본 프레젠테이션입니다 - 현재 도구 창 목록과 정렬되는 방식으로 구성된 구성표입니다. IDE(IDE 창)에서 관리하는 창의 경우 레이아웃 정보는 사용자별로 유지되므로 사용자가 IDE를 실행하면 Windows 레이아웃이 Visual Studio를 마지막으로 종료했을 때와 동일하게 나타납니다. IDE 창의 상태와 위치는 XML 형식의 사용자 지정 옵션 파일에 유지됩니다. IDE에 로드된 패키지에 의해 생성된 도구 창은 레지스트리에 해당 상태 정보를 유지하며 사용자당 상태 정보를 유지하거나 아닐 수도 있습니다.

#### <a name="profile-specific-layouts"></a>프로필별 레이아웃
 각 프로필에는 특정 개발자 가상 위치에 익숙한 방식으로 구성된 도구 창 레이아웃이 포함되어 있습니다(Visual C++ 개발자는 IDE의 왼쪽에 **솔루션 탐색기를** 볼 것으로 예상하고 C# 개발자는 오른쪽에 **솔루션 탐색기를** 볼 것으로 예상합니다). 사용자가 시작 시 프로필을 선택한 후 프로필별 창 레이아웃이 로드됩니다. 패키지 작성자가 고객의 경험에 가장 적합한 창 레이아웃을 결정해야 하며, 사용자가 창 구성에 변경한 내용은 계속 유지됩니다.

## <a name="touch-input"></a><a name="BKMK_TouchInput"></a>터치 입력
 사용자는 터치 장치에서 Microsoft 개발 제품을 점점 더 사용하고 있습니다. 그러나 터치 장치에서 개발 도구를 사용하기 어렵게 만드는 장벽이 있습니다. 사용자는 당사의 제품이 안정적이고 정밀한 터치 경험을 제공하기를 기대합니다. 이 지침의 목적은 통합할 터치 기능에 대한 결정을 알리고 Visual Studio 및 관련 제품 전반에서 일관된 터치 환경을 장려하는 것입니다.

### <a name="levels-of-experience"></a>경험 수준
 다음 수준의 경험은 팀이 원하는 수준의 터치 에 대한 투자 관심에 따라 제공할 터치 기능을 결정하는 데 도움이 되는 가이드 역할을 하기 위한 것입니다.

- **기본 환경은** 작업 전반에 걸쳐 막다른 작업이 없도록 터치 기능을 제공하려는 팀을 위한 것입니다.

- **최적화된 환경은** 가장 일반적인 터치 기능(예: 일반적으로 인터넷 브라우저 응용 프로그램에서 사용할 수 있음)을 제공하려는 팀을 위한 것입니다.

- **높은 환경은** 제스처 또는 응용 프로그램을 터치 우선 친화적으로 만들 수 있는 기타 선택적 기능과 같은 기능을 추가하려는 팀을 위한 것입니다.

||기본 경험|최적화된 경험|높은 경험|
|-|----------------------|--------------------------|-------------------------|
|사용자가 ...|막다른 끝없이 코드 및 솔루션/프로젝트 수준 읽기 수정|유지 관리, 리팩터 및 탐색 작업 수행|일관되고 직관적이며 유연한 환경에서 안심하고 운영|

|기타 고려 사항|||사용자 정의 화면 키보드|

#### <a name="gestures"></a>제스처
 제스처는 사용자에게 더 복잡한 상호 작용이 필요할 수 있는 명령에 대한 바로 가기를 제공합니다. 데스크톱 응용 프로그램에 대 한 [일반적인 터치 제스처에 대](/windows/desktop/wintouch/windows-touch-gestures-overview)한 Windows 지침을 참조 하 고 패닝 및 확대/축소 와 같은 간단한 제스처를 포함 하 여 대부분의 제스처에 대 한이 지침을 따르십시오.
