# <a name="ca1507-use-nameof-in-place-of-string"></a>CA1507: 문자열 대신 `nameof`을 사용 합니다.

|||
|-|-|
|TypeName|UseNameofInPlaceOfString|
|CheckId|CA1507|
|범주|Microsoft 유지 관리|
|주요 변경 내용|최신이 아님|

## <a name="cause"></a>원인

포함 하는 메서드의 매개 변수 이름 또는 포함 하는 형식의 속성 이름과 일치 하는 @no__t 0 리터럴 또는 상수는 메서드에 대 한 인수로 사용 됩니다.

## <a name="rule-description"></a>규칙 설명

Rule CA1507는 메서드 또는 생성자에 대 한 인수로 `string` 리터럴을 사용 하도록 플래그를 추가 합니다. 여기서 [nameof](/dotnet/csharp/language-reference/keywords/nameof) (Visual Basic의 @no__t) 식이 유지 관리를 추가 합니다. 규칙은 다음 조건이 모두 충족 되 면 발생 합니다.

- 인수는 @no__t 0 리터럴 또는 상수입니다.

- @No__t-0 리터럴의 상수 값은 다음 중 하 나와 일치 합니다.

  - 메서드 또는 생성자가 호출 되는 메서드, 람다 또는 로컬 함수의 매개 변수 이름입니다.

  - 메서드나 생성자가 호출 되는 형식의 속성 이름입니다.

- 인수는 메서드 또는 호출 되는 생성자의 @no__t 64, 형식 매개 변수에 해당 합니다. 즉, 호출 사이트와 관련 된 변환이 없습니다.

Rule CA1507는 나중에 매개 변수의 이름을 바꿀 수 있지만 `string` 리터럴의 이름이 잘못 된 경우 코드를 유지 관리 합니다. @No__t-0을 사용 하면 리팩터링 작업을 통해 매개 변수의 이름을 바꿀 때 기호의 이름이 변경 됩니다. 또한 컴파일러는 매개 변수 이름에서 맞춤법 오류를 catch 합니다.

> [!NOTE]
> 규칙 CA1507는 레거시 분석에서 사용할 수 없습니다. 이는 [FxCop 분석기](https://www.nuget.org/packages/Microsoft.CodeAnalysis.FxCopAnalyzers) 버전 2.6.0에 처음 도입 되었습니다.

## <a name="how-to-fix-violations"></a>위반 문제를 해결하는 방법

위반 문제를 해결 하려면 `string` 리터럴을 Visual Basic의 [nameof](/dotnet/csharp/language-reference/keywords/nameof) (`NameOf`) 식으로 바꿉니다. 예를 들어 다음 두 코드 조각은 규칙을 위반 하 고 해결 하는 방법을 보여 줍니다.

```csharp
public Book(string title)
{
    // Violates rule CA1507
    Title = title ?? throw new ArgumentNullException("title", "All books must have a title.");
}
```

```csharp
public Book(string title)
{
    // Resolves rule CA1507 violation
    Title = title ?? throw new ArgumentNullException(nameof(title), "All books must have a title.");
}
```

> [!TIP]
> Visual Studio에서이 규칙에 대 한 코드 수정을 사용할 수 있습니다. 이를 사용 하려면 `string` 리터럴에 커서를 놓고 **ctrl**+를 누릅니다 **.** 누릅니다. 표시 되는 옵션 목록에서 **기호 이름을 표현 하려면 nameof 사용을** 선택 합니다.
>
