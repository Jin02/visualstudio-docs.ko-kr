## <a name="BKMK_DataVisualization"></a> 데이터 시각화

### <a name="overview"></a>개요
 차트는 시각적으로 집계 하 여 의사 결정을 개선 하기 위해 데이터를 시각화 합니다. 이러한 사용자에 게 매우 주의 어떤 살펴보고 작업 필요한 권한은 거의 의미를 제외한 데이터의 많은 도움이 됩니다.

 사용자는 다음 조건 중 하나라도 해당 하는 경우 차트에서 것이 좋습니다.

- 차트를 식별할 수 있도록 작업 작업할 수 있습니다?

- 차트를 사용 하면 사용자가 잠재적인 변경에 따른 예측?

- 차트는 추세를 검색 하 고 패턴을 식별 하는 사용자가 데 도움이 됩니다.

- 차트 하면 더 나은 결정을 내릴 수 있습니까?

- 차트는 지정 된 컨텍스트에서 사용자가 있는 특정 질문에 대답 도움말은?

#### <a name="general-rules-for-charts"></a>차트에 대 한 일반 규칙

- 레이블 데이터 명확 하 게 합니다. 설명 없이 그림은 단순히 보기 좋은 그림입니다.

- 축 기울이기 비율을 방지 하기 위해 0에서 시작 합니다. 줄 길이 및 막대 크기는 데이터 요소 간의 관계를 이해 하는 데 중요 한 시각 신호입니다.

- 하지 인포 그래픽 차트를 만듭니다. 인포 그래픽의 데이터를 꾸밈 표현 되며 해당 기사의 주된 목적은 visual 스토리텔링. 차트 (하 고 수 해야) 시각적으로 뛰어난 수 있지만 자체적 데이터가 있습니다.

- Skeumorphism, 그림 막대 그래프, 대비 해시 및 기타 인포 그래픽 터치 하지 마십시오.

- 3D 효과 장식 요소로 사용 하지 마세요. 경우에 사용 하는 정보를 이해 하는 사용자의 기능을 실제로 정수입니다.

- 두 개 이상의 색 하기가 어려울 수 있습니다이 차트 종류를 읽고 올바르게 해석 하는 대로 여러 선 및 채우기, 사용 하지 마십시오.

- 차트 (또는 원하는 그림)를 개념을 이해 하 여 데이터와 상호 작용의 유일한 수단으로 사용 하지 마세요. 이 시각 장애가 있는 사용자에 대 한 문제를 표시합니다.

- 차트 페이지의 무상 또는 장식 요소로 사용 하지 마세요. 즉, 차트 하지 문제를 해결 하는 값 또는 도움말 사용자를 추가 하는 경우 해당 사용 하지 마세요.

### <a name="chart-types"></a>차트 종류
 Visual Studio에서 사용 하는 차트는 가로 막대형 차트, 꺾은선형 차트, 링 차트 또는 "도넛형 차트," 타임 라인 이라고 수정 된 원형 차트 분산형 플롯 (라고도 "차트 클러스터"), 및 Gantt 차트입니다. 각 차트 종류에 다양 한 유형의 정보를 통신할 때 유용 합니다.

### <a name="other-charting-considerations"></a>기타 차트 고려 사항

## <a name="BKMK_OnObjectUI"></a> 개체에 UI 및 보기
 이 섹션에서는 관찰, 코드 미리 보기 뷰 라고도 Visual Studio에 고유한 개체에 UI의 형식에 컨텍스트를 제공 합니다.

### <a name="overview"></a>개요

- UI 개체에 부여 해야 사용자 자세한 정보 또는 상호 작용 기본 작업에서 떨어뜨리지 않고 합니다.

- Visual Studio의 개체에 UI에 대 한 기본 패턴 "관심 지점 정보입니다." 라고 합니다.

- Visual Studio에서 UI 개체에는 인라인 또는 부동 및 하거나 durable 또는 일시적이 지입니다.

  - 코드 보기 보기, Visual Studio에서 개체에 UI 유형의 경우 인라인 및 내구성

  - CodeLens를 Visual Studio에서 UI 개체에 형식의 부동 되어 일시적인

  코드 조각의 작동 방식을 이해 하거나 해당 코드에 대 한 세부 정보를 찾으려는 컨텍스트 전환 및 기타 콘텐츠를 이동 하려면 개발자는 필요 창입니다. 해당 주 창을 떠날 경우 사용자가 자신의 원래 작업에 집중 떨어질 수 있으므로 이러한 상황에 맞는 만큼 중단을 수 있습니다. 또한 원래 컨텍스트에 다시 발생 한 다른 UI에 의해 가려질 수를 원래 코드 창 전환 하는 경우에 특히 어려울 수를 가져옵니다.

  개체에 UI "관심 지점 정보입니다." 라는 패턴을 따릅니다. 이러한 메시지, 팝업 창 및 대화 상자 사용자가 기본 작업에 계속 집중 하면서 확인 하거나 상호 작용을 추가 하는 추가 관련 정보를 제공 합니다. 개체에 UI의 예로 사용자가 알림 영역에 아이콘, 맞춤법이 틀린된 단어 및 Visual Studio 2013에 도입 된 피킹 보기에서 빨간색 오류 표시선 위로 마우스 포인터를 가리킬 때 나타나는 팝업 창이 있습니다.

### <a name="decision-points"></a>의사 결정 요점
 Visual Studio 내에서 주의가 시점 정보의이 패턴을 사용 하는 방법은 여러 가지가 있습니다. 올바른 메커니즘을 선택 하 고 일관 되 고 예측 가능한 방식으로 구현 전체 환경에 필수적입니다. 이 고, 그렇지 사용자 콘텐츠 자체에서 포커스에 혼동을 줄 또는 일관 되지 않은 환경으로 표시 될 수 있습니다.

#### <a name="relationships-between-master-and-detail-content"></a>마스터 및 콘텐츠 세부 정보 간의 관계
 관심 지점 정보 관계를 표시 하는 데 사용 됩니다 관련 콘텐츠 ("정보" 콘텐츠)를 ("master" 콘텐츠)에 집중 하 고 추가 사용자가 콘텐츠 사이입니다. 이 패턴에서는 정보 콘텐츠는 콘텐츠는 사용자가 사용 및 마스터 콘텐츠 가까운 표시 될 수 있습니다 명확 하 게 관련 되어 있습니다. 추가 정보 또는 마스터 콘텐츠 부담 없이 요약할 수 없는 정보 도구 창과 같은 다른 패턴을 따라야 합니다.

- **항상** 근접 마스터 콘텐츠 정보 콘텐츠를 표시 합니다.

- **항상** 세부 내용을 마스터 콘텐츠에 계속 집중 하는 사용자는 여전히 활성화 되어 확인 합니다. 대개이 작업을 수행 하는 가장 좋은 방법은 마스터 콘텐츠 최대한 가까운으로 세부 내용을 렌더링 하는 합니다. 마스터 콘텐츠 옆의 팝업 창에서 세부 정보 콘텐츠를 렌더링 하 여 마스터 콘텐츠 아래 세부 정보 콘텐츠 인라인을 렌더링 하 여 수행할 수 있습니다.

- **되지** 주의 마스터 콘텐츠에서 사용자를 사용 하는 시점에서 정보를 사용 합니다. 사용자를 개별적으로 세부 내용을 확인 해야 하는 경우 이렇게 할 수 있도록 하는 명시적인 작업을 노출 합니다.

#### <a name="design-details"></a>디자인 정보
 UI 개체에 올바른 선택 인지 결정 한 경우에 네 가지 주요 디자인 고려 사항

1. **지 속성:** 콘텐츠 이어야 하는데 지속적인 지 또는 일시적인?
   사용자가 됩니다 계속 정보를 참조 하거나 상호 작용을 표시 하 시겠습니까? 또는 사용자가 해야 할 정보를 신속 하 게 살펴봅니다 하 고 해당 기본 작업을 사용 하 여 계속?

2. **콘텐츠 형식:** 정보, 실행 가능한, 또는 탐색 콘텐츠 수는?
   사용자 마스터 콘텐츠에 대 한 추가 세부 정보에 필요 합니까? 사용자는 마스터 콘텐츠에 영향을 주는 작업을 완료 해야 합니까? 또는 사용자 않아도 다른 리소스에 연결할 수 있습니까?

3. **표시기 유형:** 앰비언트 표시기 합리적?
   정보 요약을 유용한 방식으로 하 수 마스터 콘텐츠 부담 없이 표시?

4. **제스처:** 제스처 어떤 사용할 호출 하 고 UI를 해제 하 시겠습니까?
   사용자 세부 내용을 표시 하 고 어떻게 번 보내도? 가치가 있는 임시 및 영구 상태 간에 전환 하려면 고정와 같은 제스처가 추가?

   개체에 UI의 주요 구성 요소에 영향을 미칠는 각각의 이러한 네 가지 의사 결정 지점.

#### <a name="container-content-presenter-type"></a>컨테이너 (콘텐츠 프레 젠 터) 형식
 가지 주의 지점에서 콘텐츠를 제공 하는 사용 가능한 두 가지 주요 옵션이 있습니다.

1. **인라인:** 는 인라인 프레 젠 터를 Visual Studio 2013 코드 편집기에서 도입 된 피킹 보기와 같은 기존 콘텐츠를 이동 하 여 새로운 콘텐츠에 대 한 공간을 만듭니다.

    - **원하는** 제공할 사용자 시간 참조 나 콘텐츠를 사용 하 여 상호 작용 투입 해야 할 경우 인라인 프레 젠 터 있습니다.

    - **방지** 사용자가 예상 하는 경우 인라인 프레 젠 터 있는 다음 해당 주 작업 중단을 최소화 하면서 진행 정보를 한 눈에 보기 하려고 합니다.

2. **부동:** 부동 프레 젠 터가 선택된 된 콘텐츠를 최대한 가깝게 배치 하지만 기존 콘텐츠의 레이아웃을 변경 하지 않습니다. 위에 떠 있는 콘텐츠 패널을 표시 하는 등 다양 한 전략을 사용을 수 있습니다는 공백을 선택한 기호를 사용할 수 있는 가장 가까운 합니다.

    - **원하는** 있는 다음 해당 주 작업 중단을 최소화 하면서 진행 정보를 한 눈에 보기 해야 할 사용자 경우 프레 젠 터를 부동 합니다.

    - **방지** 시간 참조 나 콘텐츠를 사용 하 여 상호 작용 투입 해야 할 사용자를 예상 하는 경우 프레 젠 터를 부동 있는 있습니다.

#### <a name="content-type"></a>콘텐츠 형식
 세 가지 주요 유형의 모든 개체에 UI 컨테이너 내에서 표시 될 수 있는 콘텐츠 있습니다. 정보의 이러한 유형의 모든 조합을 표시할 수 있습니다. 세 가지 형식은 다음과 같습니다.

1. **알림:** 대부분 개체에 UI 컨테이너 일부 종류의 정보 콘텐츠를 표시 합니다. 잠재적 향후 상태의 환경에 대 한 정보를 나타낼 수 또는 콘텐츠 환경의 현재 상태에 대 한 정보를 나타낼 수 있습니다. 예를 들어, 기존 코드를 리팩터링 하는 등 특정 명령의 효과 보여 주도록 사용할 수 없습니다.

    - **항상** 수를 표시 하는 정보의 정식 표현을 사용 합니다. 예를 들어, 코드 구문 강조 표시 된 전체 코드와 같습니다 및 글꼴 및 기타 환경 설정을 사용자가 설정한 고려해 야 합니다.

    - **항상** 수 동일한 정보에는 마스터 콘텐츠도 표시 되 면 정보 내용에 대 일체를 지 원하는 것이 좋습니다. 예를 들어 있는 개체에 UI 컨테이너 내에서 기존 코드를 표시 하는 경우 좋습니다 검색 하 고 해당 코드를 수정할 수 있는 기능을 지원 합니다.

    - **항상** 잠재적인 향후 상태를 나타내며는 정보 제공 용 이므로 콘텐츠를 제공 하는 경우 다른 배경색을 사용 하는 것이 좋습니다.

2. 실행 가능한: 일부 개체에 UI 컨테이너 리팩터링 작업을 수행 하는 등 마스터 콘텐츠를 통해 일부 작업을 수행 하는 기능을 제공 합니다.

    - **항상** 정보 콘텐츠를 별도로 실행 가능한 명령을 배치 합니다.

    - **항상** 사용 하도록 설정 하 고 해당 하는 경우 작업을 사용 하지 않도록 설정 합니다.

    - **항상** 대화 상자 내에서 명령을 나타내는 표준의 지침을 참조 하세요.

    - **항상** 최소 절대 수 있는 개체에 UI 컨테이너에 노출 되는 작업의 수를 유지 합니다. 개체에 UI와 상호 작용을 간단 하 고 빠른 환경 이어야 합니다. 사용자 개체에 UI 컨테이너 자체 관리에서 에너지를 사용할 필요가 없습니다.

    - **항상** 개체에 UI 컨테이너를 종료 되거나 해제 될 방법 및 시기를 고려해 야 합니다. 모범 사례로, 마스터 및 콘텐츠 정보 간의 대화를 완료 하는 모든 작업도 함께 닫을지 UI 컨테이너 개체에 해당 작업을 호출할 때.

3. **탐색:** 일부 개체에 다른 창이 나 사용자의 웹 브라우저에서 MSDN 문서를 열기와 같은 응용 프로그램에 사용자를 사용 하는 링크를 포함 하는 UI 컨테이너입니다.

    - **항상** 다른 콘텐츠를 탐색 하 여 사용자가 당황 하지 않을 수 있도록 "열기"를 사용 하 여 탐색 링크 앞에 추가 합니다.

    - **항상** 실행 가능한 링크의 탐색 링크를 구분 합니다.

#### <a name="ambient-indicators-optional"></a>앰비언트 표시기 (선택 사항)
 코드의 나머지 부분과에서 대비 색에서 제시 된 텍스트를 포함 하 여 미묘 또는 물결선 밑줄 등 스마트 태그 아이콘 tickler 기호를 포함 하 여 확실 한 앰비언트 표시기 수 있습니다. 앰비언트 표시기 추가, 관련 정보의 가용성을 통신합니다. 이상적으로 사용자 상호 작용 하도록 요구 하지 않더라도 유용한 정보를 제공 합니다.

- **항상** 방해가 하거나 사용자에 과부하가지 않습니다 있도록 앰비언트 표시기를 배치 합니다. 이러한 방식으로 앰비언트 표시기를 배치 하는 것이 불가능 합니다 하는 경우 다른 솔루션을 고려 합니다.

- **항상** 앰비언트 표시기와 관련 된 내용에 최대한 가깝게 배치 합니다.

- **항상** 사용할 수 있는 정보를 요약 하는 표시기를 만들려고 시도 합니다. 수가 사용 가능한 데이터 항목의 수 (예를 들어 "3 참조" 단순히 "참조" 대신)를 제공 하는 것이 좋습니다. 또는 데이터를 요약 하려면 다른 방법으로 생각할 수 있습니다.

    - 위치 표시기에 대 한 일 수 없으며 항상 계산을 표시의 경우 즉시 계산 되 고 값으로 점진적 의견 고려 합니다. 예를 들어, 읽지 않은 메일 증가 수와 Windows Phone 전자 메일 라이브 타일을 새로 고칩니다 방식과 유사 하 게 사용할 수 있는 데이터에 대 한 업데이트를 반영 하는 변경에 애니메이션을 적용 하는 것이 좋습니다.

- **되지** 사용자 지정된 콘텐츠의 합리적으로 수행할 수 있는 보다 자세한 표시기를 추가 합니다. 앰비언트 표시기 사용자에서의 개입 없이 유용 합니다. 오버플로 및 기타 관리 컨트롤 보기에 가져오는 것이 필요할 경우 해당 앰 비 언스를 손실 하는 표시기.

#### <a name="gestures"></a>제스처
 열고 추가 세부 정보 콘텐츠를 해제 합니다. 오른쪽 제스처를 지원 하 여 마스터 콘텐츠에 초점을 유지 하기 위해 사용자의 주요 측면은입니다.

- **항상** 추가 콘텐츠를 열기 일부 명시적 제스처를 수행 하는 사용자에 게 요구 합니다. 일반적인 오픈 제스처는 다음과 같습니다.

    - **Hover:** 도구 설명 또는 비 대화형 정보 콘텐츠

    - **명시적인 명령:** 인라인 프레 젠 터

    - **앰비언트 표시기를 두 번 클릭 합니다.** CodeLens 팝업 창

- **항상** Esc 키를 누를 때마다 정보 콘텐츠를 해제 합니다.

- **항상** 개체에 UI의 상황을 고려 하세요. 컨테이너 내에서 상호 작용할 수 있도록 콘텐츠 발표자 신중 하 게 시간적 고려 마우스로 가리키면 추가 정보를 표시 하는 경우 사용자의 워크플로를 중단 될 수 있습니다.

- **되지** 편집 가능 보기가 표시 하거나 사용자 상호 작용을 초대 하는 hover에 콘텐츠를 표시 합니다. 커서를 가져갈 때 더 이상 마스터를 통해 생성 한 콘텐츠를 즉시 해제 하는 도구 설명에 대 한 표준 동작 세부 정보 콘텐츠를 위로 커서를 이동 하 려 할 경우이 동작은 사용자가 불편 해질 수 없습니다.

## <a name="BKMK_SelectionModels"></a> 모델 선택

### <a name="overview"></a>개요
 선택 모델은 나타내며 사용자 인터페이스 내에서 필요한 하나 이상의 개체에 대 한 작업을 확인 하는 데 사용 하는 메커니즘입니다. 이 항목에서는 Visual Studio 문서 편집기 내에서 선택 상호 작용 패턴을 설명 합니다: 텍스트 편집기, 디자인 화면 및 모델링 화면.

 사용자에 맞게 작업할 Visual Studio로 나타낼 수 있어야 합니다. 및 Visual Studio에 응답 해야 예측 가능한 방식으로 피드백을 사용 하 여 운영에 대 한 사용자. 차이나는 잘못 된 사용자와 사용자 인터페이스 간의 통신 하지 수 있는 액션을 확인 하면 발생할 수 있습니다 의도 하지 않은 결과가 발생 합니다. 종종 것이 없거나 변경 된 사용자에 게 표시 될 때까지 오류 알려지지 합니다. 모델 선택 되므로 사용자 인터페이스 디자인의 가장 중요 한 부분 중 하나입니다. Visual Studio에서 선택 모델은 Windows와 일치 하지만 약간의 변형이 있습니다.

 Windows에서와 같이 Visual Studio에서 선택 모델 상호 작용이 발생 하는 컨텍스트에 따라 달라 집니다. 선택 항목은 네 가지 유형의 개체에서 발생할 수 있습니다.

- 텍스트

- 그래픽 개체

- 목록 및 트리

- 표

  이러한 개체에는 세 가지 유형의 선택

- 연속

- 서로 분리

- Region

#### <a name="scope"></a>범위
 선택의 가장 중요 한 구성 요소 확인 하는 것 (활성화)는 창의 작업 및 배치 (선택)를 포커스가 있는 사용자가 알고 있는 합니다. Visual Studio, Windows의 창 관리 기능을 확장 하지만 정품 인증 체계는 동일 합니다: 창으로 포커스를 창과 상호 작용 합니다. Visual Studio에는 정품 인증에 대 한 두 가지 지표: 문서 창 및 도구 창에 대 한 합니다.

 문서 창에 대 한 활성 창 앞에 오는 배경 색을 변경 하는 문서 창의 탭에 표시 됩니다.

 도구 창에 대 한 활성 창 도구 창의 제목 표시줄 영역 색을 변경 하 여 표시 됩니다.

 ![Visual Studio에서 활성 도구 창 선택 영역](../../extensibility/ux-guidelines/media/0713-02-activetoolwindow.png "0713 02_ActiveToolWindow")

 **노드의 기본 선택 영역을 표시 하는 활성 도구 창**

 **노드의 잠재 선택 영역을 표시 하는 비활성 도구 창**

 창이 활성화 되 면 해당 포커스는이 섹션의 지침에 설명 된 선택 모델에 따라 표시 됩니다.

#### <a name="context"></a>컨텍스트
 Visual Studio 사용자가 작업 하는 위치를 추적 하는 강력한 개념이 컨텍스트를 유지 하도록 설계 되었습니다. 하나의 기간은 도구 또는 문서 창 인지 활성 상태입니다. 그러나 최상위 문서 창에는 잠재 선택 항상 유지 됩니다. 포커스가 도구 창에 있을 수 있습니다, 있지만 마지막으로 활성 문서 창을 선택 비활성 상태에도 표시 합니다. 사용자의 컨텍스트에서 된 편집 문서를 반환 하 고 도구 창과 문서 창 간에 원활 하 게 이동할 수 있도록 Visual Studio에서 해당 상태를 유지에 표시를 유지 하려면 수행 됩니다.

### <a name="text-selection"></a>텍스트 선택
 에 설명 된 모양 및 동일한 텍스트 선택 모델을 사용 하는 기본 제공 텍스트 편집기와 같은 텍스트 엄격 하 게 된 visual Studio 편집기는 [마우스 및 포인터](https://msdn.microsoft.com/library/dn742466.aspx) 페이지에서 Windows 사용자 환경 상호 작용 지침 MSDN입니다. 텍스트 편집기에서 입력된 포커스를 삽입 지점 이라고 함 세로 막대로 표시 됩니다. 삽입 지점이 있을 굵은 및 그 뒤에 모든 표시의 역으로 색이 지정 된 단일 픽셀입니다. 설정한 속도 따라 깜박입니다를 **커서 깜박임 속도** 에서 설정를 **속도** 탭의 **키보드** 제어판에 애플릿을 합니다.

#### <a name="contiguous-and-disjoint-selection"></a>연속 및 비연속 선택
 텍스트 편집기에서 선택한 연속 된다입니다. 비연속 텍스트 선택을 허용 되지 않습니다 하지만 그래픽 개체 편집기에서 해결 해야 합니다. 사용자의 마우스 포인터가 위에 텍스트 영역, i 형 커서 변경 됩니다. 한 번의 클릭 클릭 위치에서 텍스트 편집기에서 삽입 지점을 넣습니다. 마우스 단추를 누른 채 선택 강조 표시를 시작 하 고 강조 표시 선택 영역이 종료 마우스 단추에서 손을 떼기.

#### <a name="region-selection-box-selection"></a>지역 선택 (상자 선택)
 Visual Studio 텍스트 편집기에서 지역 선택 항목을 지원 하 고 상자 선택 이라고 합니다. 상자 선택 하면 일반 텍스트 스트림에 따르지 않는 텍스트 영역을 선택 합니다. 표준 텍스트 선택, 마찬가지로 선택이 연속적 이어야 합니다. 상자 선택 Alt 키를 누른 채 마우스를 사용 하 여 드래그 하 여 시작 됩니다. 또한 Alt 및 Shift 키를 누른 채 선택 영역을 가리키는 화살표 키를 사용 하 여 상자 선택을 시작할 수 있습니다. 상자 선택 일반 선택 강조 표시를 사용 하 고 선택 영역 끝에 깜박이 삽입 지점 커서를 보여 줍니다.

#### <a name="text-selection-appearance"></a>텍스트 선택 항목 모양
 편집기에서 활성 및 비활성 선택 영역에 사용 된 색을 사용자 지정할 수 있습니다. 편집기의 시각적 모양을 사용자 지정 사용자 수로 이동 **도구 > 옵션**를 찾은 후 아래 **환경 > 글꼴 및 색 > 텍스트 편집기**합니다.

### <a name="graphical-selection"></a>그래픽 선택

#### <a name="interaction"></a>상호 작용
 그래픽 개체 선택 복잡할 수 있으며 여러 가지 요인에 따라 달라 집니다.

- **편집기의 기본 선택 모델입니다.** 그래픽 개체를 포함 하는 편집기는 텍스트 또는 표를 편집 하려면 데도 사용할 수 있습니다. 예를 들어, 편집기에도 Visual Studio XAML 디자이너와 같은 그래픽 개체의 배치를 지 원하는 텍스트 기반 편집기를 수 있습니다. 여러 개체 형식을 지 원하는 사용자가 다양 한 유형의 개체에 구성 된 그룹을 선택 하는 방법에 발생할 수 있습니다.

- **기본 및 보조 선택 상태에 대 한 지원입니다.** 편집기 상태 개체를 한꺼번에 편집할 수 있도록 서로 맞춰져, 함께 등과 크기를 조정 하는 기본 및 보조 선택 항목을 제공할 수 있습니다.

- **내부 편집 지원입니다.** 편집기에 편집 된 그래픽 개체의 콘텐츠를 사용할 수 있습니다. 예를 들어 사각형 모양을 사용자가 변경할 수 있는 내부에 텍스트를 포함할 수도 있습니다. 또한 텍스트 가운데 또는 양쪽 수 없습니다. 자세한 수준의 사용자 상호 작용을 포함 하 고 적절 한 시각 신호 집합을 사용자에 게 상태 정보를 제공 하므로 전체 편집입니다.

#### <a name="tree-view"></a>트리 뷰
 트리 뷰에서 선택으로 간단한 강조 표시 됩니다. 노드 이름 또는 노드 아이콘에서 사용자가 노드 선택 됩니다. 노드의 왼쪽에 있는 삼각형 문자 모양을 확장 또는 트리 컨트롤 계약 하지만 한 가지 예외를 사용 하 여 사용자의 선택에 영향을 주지 않습니다: 부모를 하나씩 선택 시 해당 노드의 자식 항목을 선택 하면 부모 노드를 축소 합니다.

 ![Visual Studio에서 일반적인 트리 뷰](../../extensibility/ux-guidelines/media/0713-25-treeview.png "0713 25_TreeView")

 **Visual Studio에서 일반적인 트리 뷰**

 트리 보기는 트리의 여러 수준 에서도 연속 및 비연속 선택 항목을 지원할 수 있습니다. 연속 또는 연결 되지 않은 다중 선택을 표시 되는 트리 노드의 수 있어야 합니다. 노드를 축소 하는 경우 연결 되지 않은 선택 손실 되 고 축소 된 노드 선택 항목을 가져옵니다. 이러한 방식으로 사용자 영향을 받는 작업에 의해 노드를 확인할 수 있습니다. 노드가 축소 되 면 명확 하지 않은 노드 영향을 줄 수 있습니다.

 부모 노드가 선택 될 때 부모 및 모든 자식에 적용 하는 작업에 대 한 적절 한 경우가 있을 경우 부모 작업을 적용 해야 합니다. 이 경우 확인란 또는 사용자에 게 "모든 자식 항목에 적용" 옵션을 명시적으로 만들어야 합니다 확인 대화 상자와 같은 작업을 하는 동안 추가 UI를 제공 합니다.

- 중첩 된 계층 구조를 포함 하는 눈금의 경우 오른쪽 화살표 키를 부모 행를 확장 하 고 왼쪽 화살표 키 하나를 축소 합니다.
